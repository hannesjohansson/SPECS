# Reference

Formats, schemas, lifecycles, and operational mechanics for the spec-driven operating model. The **why** — philosophy, principles, and contrasts with other approaches — lives in [operating-model.md](operating-model.md). This doc is the companion: it tells you exactly how each artifact is shaped and how the system runs day to day.

> Examples throughout use a **personal todo app** as a concrete illustration of an otherwise domain-agnostic model. Swap `task`, `list`, `daily review`, etc. for whatever your product actually is.

## How it works end-to-end

The system is a closed loop, from strategic claim at the top to durable evidence at the bottom. Here's a single walkthrough using the daily-review example:

1. **A strategic claim is recorded as a `bet`** — "daily review turns task backlog into a trusted system." Hypothesis, kill criteria, metrics, signals. Drafted via the [`draft-bet`](../.claude/skills/draft-bet/SKILL.md) skill; lands on the `plans` branch.
2. **A tactical pilot is drafted as an `initiative`** — "daily-review pilot: end-of-day prompt, two-week window." The initiative cites the bet(s) it tests, carries its own falsifiable hypothesis and kill criteria, and names the specs it will touch. Drafted via [`draft-initiative`](../.claude/skills/draft-initiative/SKILL.md); lands on `plans`.
3. **The initiative's first-touch check** runs for each spec it names: *does the spec exist?* If not, *does the behavior already exist in code?* "Behavior exists, no spec" ⇒ queue a codifying `draft-spec` task before any change tasks. See [operating-model.md § First touch of an un-specced behavior](operating-model.md#first-touch-of-an-un-specced-behavior).
4. **Specs are drafted as three-layer artifacts** — `## Narrative` (explanatory), `## Rules` (machine-authoritative contract), `## Checks` (one enforced per binding rule = regression suite by construction). Drafted via [`draft-spec`](../.claude/skills/draft-spec/SKILL.md); updated via [`update-spec`](../.claude/skills/update-spec/SKILL.md); lives on `main`. Specs carry no hypothesis — the why lives one layer up, on the initiative.
5. **Code is implemented** against a spec's binding rules via the [`implement`](../.claude/skills/implement/SKILL.md) skill. `/validate` checks schema; `/drift` checks that code matches rules. At least one enforced check per binding rule is required. Lands on `main` via PR.
6. **Tasks are the execution unit.** Every step 2–5 happens through a claimed task on the `plans` branch: `/work` → `/claim <id>` → do the work on a feature branch → PR → `/complete <id>` → `/verify <id>` (by a different identity).
7. **Insights close the loop.** When the initiative's pilot concludes, [`write-initiative-insight`](../.claude/skills/write-initiative-insight/SKILL.md) records `confirmed | refuted | inconclusive`. If the evidence materially moves the bet, [`write-bet-insight`](../.claude/skills/write-bet-insight/SKILL.md) fires too. Insights never edit spec rules directly — humans decide the next move based on the evidence.

Two measurement layers run in parallel: *did this pilot pay off?* (initiative insight) and *is the strategic bet holding up?* (bet insight). Both inform what gets drafted next.

## Cross-document navigation

YAML frontmatter is the source of truth for references (`bets`, `specs_addressed`, `initiative`, `updates_spec`, `measures_bet`, `measures_initiative`, etc.). Tooling (`validate`, `drift`, `graph`, `queue`) reads those fields. Most scripts run from the plans worktree — the sparse-checkout materializes `specs/` and `direction/insights/` read-only so cross-references resolve without leaving the worktree.

Navigation for humans lives in one place: [`INDEX.md`](../INDEX.md), regenerated by `bun run indexes` from the same frontmatter. Bets, initiatives, and specs do not hand-maintain `## References` or `## Referenced by` sections — frontmatter plus the generated index cover it. Inline prose may link other IDs when the mention is load-bearing; IDs remain canonical, paths are navigation only.

## Config

```yaml
# process/config.yaml
mode: pre-traction   # or: post-traction
plans_branch: plans  # branch that orchestration commits (bets, initiatives, task state) must run on
stale_claim_threshold: 4h
```

### Modes

One flag in `config.yaml` drives mode-specific rigor:

**Pre-traction** (no real users, no baseline data): hypothesis + kill criteria required on bets and active initiatives; metrics accept `unknown` as current value; insights accept `preliminary` status.

**Post-traction** (real users, real data): everything above, plus stories extracted and validated; product initiatives must reference at least one story; metrics must have real current values; insights must cite data.

`validate` enforces the active mode. Flip to `post-traction` once you have enough users to measure real metrics.

### North star

`direction/north-star.md` — plain markdown, no frontmatter. One paragraph naming the single metric the company is aiming at, with a sentence on why. Every bet either grows, protects, or lays foundation for that number.

## Artifacts

Artifacts form layers, each serving a different timescale and carrying a different kind of authority. Strategic claims live the longest; execution tasks are ephemeral. See [operating-model.md § The artifacts](operating-model.md#the-artifacts) for the conceptual model; the reference below is the shape.

| Layer | Artifact | Owns | Lives | Authority |
|---|---|---|---|---|
| Strategic | [Bet](#bet) | hypothesis, kill criteria, metrics | `planning/bets/` on `plans` | predictive (why to invest) |
| Tactical | [Initiative](#initiative) | pilot hypothesis, scope, `specs_addressed` | `planning/initiatives/` on `plans` | predictive (why this pilot) |
| Tactical | [Story](#story) | user's when/want/so-that statement | `planning/stories/` on `plans` | framing (what users need) |
| Contract | [Spec](#spec) | narrative + rules + checks | `specs/[domain]/` on `main` | authoritative (code must match rules) |
| Contract | [Rule](#rule-inline-and-shared) | one binding statement | inline in spec, or `shared/rules/` | authoritative |
| Contract | [Check](#check) | mechanical verification of one rule | `specs/[domain]/checks/` | mechanical (regression substrate) |
| Execution | [Task](#task) | one claimable unit of work | `planning/tasks/<state>/` on `plans` | procedural |
| Evidence | [Insight](#insight) | `confirmed/refuted/inconclusive` verdict | `direction/insights/` on `main` | durable evidence |

Ownership of branches: see [Git coordination](#git-coordination).

### Bet

A strategic claim the team is investing in. Carries a durable hypothesis that spans multiple tactical initiatives; a bet closes when its hypothesis is answered (or paused / abandoned), not when a particular piece of work ships.

**Frontmatter** (reference-shaped; prose in the body):

```yaml
id: bet-daily-review
type: bet
status: active
title: Daily review turns task backlog into a trusted system
connected_stories: []
initiatives: [init-daily-review-pilot]
updated: 2026-04-19
```

**Body sections (in order):** `## Hypothesis`, `## Kill criteria`, `## Metrics`, `## Signals`, `## Why this bet`, `## What success looks like`, `## Known risks`, `## Deferrals`.

Example body opening:

```markdown
# Bet: Daily review

## Hypothesis

A personal task list stays trusted only if overdue items get triaged frequently. If an end-of-day prompt nudges users to either reschedule or drop overdue tasks, the backlog stays small enough to act on; if the prompt fails to turn pile-up into action, the list is avoided and the product loses.

**Confidence:** medium — Users already re-triage manually on Mondays, so the concept isn't foreign. We haven't validated that a daily prompt beats ad-hoc cleanup.

## Kill criteria

If across two pilots users consistently dismiss the review prompt without acting (reschedule, drop, or complete), the approach is miscast.

## Metrics

- **Active lists per user (median)** — current: unknown, source: platform
  Core engagement number. Establish baseline in first pilot.
- **User-reported trust in their list (1–5)** — current: unknown, source: survey
  Preserves the quality side of the trade.

## Signals

- Users report spending less time worrying about forgotten tasks.
- Users report the list "feels like it's theirs" — overdue count stops being a source of dread.
```

**Lifecycle:** `backlog → planned → active → completed`. `backlog` is framing present but not scheduled; `planned` means the bet is next up (initiatives being drafted against it); `active` means at least one initiative is running against it; `completed` means the team has stopped investing. Insights capture the outcome; state captures work-phase.

**References and flow:**
- Tested by initiatives (each initiative lists `bets: [...]`, M:N).
- Measured by insights with `measures_bet: <bet-id>`.
- Drafted via [`draft-bet`](../.claude/skills/draft-bet/SKILL.md); closed with a [`write-bet-insight`](../.claude/skills/write-bet-insight/SKILL.md) entry.

### Initiative

A tactical pilot under one or more bets (M:N). Carries a falsifiable hypothesis scoped to this pilot, its own kill criteria, success criteria, and the specs it commits to. One initiative = one type (`product | architecture | fix | discovery`). Mixed-type pilots split.

**Frontmatter:**

```yaml
id: init-daily-review-pilot
type: initiative
initiative_type: product            # product | architecture | fix | discovery
status: active
title: Daily-review pilot — end-of-day prompt, two-week window
bets: [bet-daily-review]             # M:N — list of bets this initiative tests
success_criteria:
  - "≥40% of enrolled users complete at least 5 daily reviews in the 2-week window."
  - "Reviewing a task does not mark it done."
specs_addressed: [spec-task-reschedule, spec-daily-review]
stories_addressed: [story-reschedule-task]   # product initiatives
# architecture_refs: []             # architecture initiatives
# reveals_gap_in: [spec-*]          # required for fix initiatives
window_hint: "2 weeks"
insights: []
updated: 2026-04-19
```

**Body sections (in order):** `## Hypothesis`, `## Kill criteria`, `## Why this is one initiative`, `## Pilot scope`, `## How each spec fits`, `## Deferrals`. Hypothesis ends with `**Confidence:** low|medium|high — {basis}`.

**Type-specific fields:**

| Type | Required fields | Notes |
|---|---|---|
| `product` | `stories_addressed` (post-traction) | user-facing behavior change |
| `architecture` | `architecture_refs` | system property improvement |
| `fix` | `reveals_gap_in: [spec-*]` (non-empty) | replaces the old fix-spec artifact; tasks `updates_spec` the target(s) directly |
| `discovery` | none (`specs_addressed: []`; set `window_hint`) | time-boxed investigation that emits an insight instead of code |

**`## How each spec fits`** must name the expected rule delta per addressed spec — *rules to add*, *rules to retire (with what supersedes them)*, *rules to preserve*. Reviewers use this to see the regression blast radius at initiative-review time.

**First-touch check.** For each spec id in `specs_addressed`: if the spec doesn't exist but the behavior already exists in code, the first task in the queue must be a codifying `draft-spec` task. See [operating-model.md § First touch of an un-specced behavior](operating-model.md#first-touch-of-an-un-specced-behavior).

**Lifecycle:** `backlog → planned → active → completed`. Same states as Bet but **separate type** so the two can diverge later without a shared-enum migration. `completed` means the pilot has concluded; the insight says whether it was a win.

**References and flow:**
- Parents tasks (each task's `initiative: <init-id>`).
- Tests bets (each initiative lists `bets: [...]`).
- Addresses specs (via `specs_addressed`); specs do **not** back-reference the initiative.
- Measured by insights with `measures_initiative: <init-id>`.
- Drafted via [`draft-initiative`](../.claude/skills/draft-initiative/SKILL.md); closed with [`write-initiative-insight`](../.claude/skills/write-initiative-insight/SKILL.md).

### Story

A user's when-I-want-so-I-can statement, extracted when the same need appears in multiple initiatives. Required post-traction for product initiatives.

**Frontmatter:**

```yaml
id: story-reschedule-task
type: story
status: validated
journey_phase: review
refs_insights: [insight-overdue-task-causes]
```

**Body:**

```markdown
**When** I open my list and see overdue tasks I couldn't get to,
**I want to** reschedule them to a realistic date,
**so I can** trust my list instead of ignoring it.
```

**Lifecycle:** `draft → validated → addressed → retired`.

Stories link to initiatives (via `stories_addressed` on product initiatives), not directly to specs.

### Spec

The smallest authoritative coordination unit for a code contract — a narrow, durable slice of behavior (roughly the size of a feature). Three layers: **narrative** (explanatory), **rules** (machine-authoritative), **checks** (regression substrate). A spec persists across many iterations via rule-level lifecycle; it is *not* replaced per feature. See [operating-model.md § Spec granularity and decomposition](operating-model.md#spec-granularity-and-decomposition).

**Frontmatter:**

```yaml
id: spec-task-reschedule
type: spec
status: accepted                   # draft | accepted | retired
title: Task reschedule
journey: list/review
```

**Body sections (in order):** `## Narrative`, `## Rules`, `## Checks`, `## Deviations`, `## Open questions`.

Example body:

```markdown
## Narrative

A reschedule changes the due date of an existing task. Past dates are rejected;
saves are atomic; the original task is unchanged on failure. Task *creation*
is covered by `spec-task-create`; *deletion* is covered by `spec-task-delete`.
This spec covers only the change-the-due-date slice.

## Rules
<!-- Authoritative. Code must match. -->

### Reschedule target
A reschedule sets a new due date on an existing task. The new date must be today or later; past dates reject.
Status: binding

### Atomic edits
Reschedules are atomic on save; partial saves reject. A failed save leaves the task unchanged.
Status: binding

## Checks
<!-- One enforced check per binding rule — the regression suite for this spec. -->

- `check-task-reschedule-target` (enforced) — verifies the reschedule-target rule.
- `check-task-reschedule-atomic-edits` (enforced) — verifies atomic edits.

Runners live under `specs/list/checks/`.

## Deviations

_None yet._

## Open questions
<!-- Unresolved ambiguity during drafting lives here, not as vague prose in rules.
     Empty or explicitly deferred (with linked follow-up) at acceptance. -->

_None._
```

**The triad.** Narrative is explanatory and updated to match rules (if they conflict, rules win). Rules are the contract — each has a status (`proposed → binding → retired`) and code must match rules at `binding`. Checks mechanically verify rules — retiring a rule retires its check; adding a rule adds a check.

**Neutrality on existing-vs-new.** A spec may codify behavior that already exists, describe new behavior, or both. Rule status distinguishes them mechanically; no separate flag.

**`## Open questions`** is the home for ambiguity surfaced during drafting — numbered, one sentence each. Ambiguity lives here, not as vague rules or placeholder prose.

**Acceptance preconditions.** A spec may enter `status: accepted` only when:
1. `## Narrative` is present and non-empty.
2. At least one rule is at `binding` (all-`proposed` specs remain `draft`).
3. Every `binding` rule has at least one `enforced` check, or a recorded deviation explains why not.
4. `## Open questions` is empty, or each remaining question is explicitly deferred with a linked follow-up task.

`accepted` means a usable regression contract, not a partially-baked intent.

**Frontmatter hygiene.** Specs carry no hypothesis, no kill criteria, no stories, no goal links, and no back-reference to an initiative. The relationship flows through the initiative's `specs_addressed` list and tasks' `updates_spec` field. The hypothesis about *why* the rules are right lives on the initiative.

**Lifecycle:** `draft → accepted → retired`. `accepted` means rules are authoritative, drift enforces code-match continuously, and the check set is the regression suite. There is no `shipped` state — with no release primitive, "is code in sync?" is a drift check, not a status field.

**References and flow:**
- Addressed by initiatives (via `specs_addressed`); specs themselves don't back-reference.
- Modified by tasks with `updates_spec: <spec-id>`.
- Drafted via [`draft-spec`](../.claude/skills/draft-spec/SKILL.md); updated via [`update-spec`](../.claude/skills/update-spec/SKILL.md); implemented against via [`implement`](../.claude/skills/implement/SKILL.md).

### Rule (inline and shared)

A rule is one authoritative statement a spec makes. Rules have a lifecycle independent of the spec that contains them: `proposed → binding → retired`. Only `binding` rules are enforced; `retired` rules are historical.

**Inline rules** live in a spec's `## Rules` section:

```markdown
### Reschedule target
A reschedule sets a new due date on an existing task. The new date must be today or later; past dates reject.
Status: binding
```

**Shared rules** live at `shared/rules/<id>.md` and list the specs they apply to:

```yaml
id: rule-api-response-format
type: rule
status: binding
title: API response format standard
applies_to: [spec-task-create, spec-task-reschedule]
```

**Extraction trigger:** when the same rule appears in two specs, lift it to `shared/rules/`. Don't extract speculatively — wait for the duplication to become real (see [Extraction pattern](#extraction-pattern)).

**Drift contract on binding rules.** Non-additive changes (retiring or narrowing a `binding` rule) must not silently rewrite the rule — either mark it `retired` with `replaced_by: <new-rule-id>` and add a superseding rule, or supersede with a new rule at `proposed`. Silent in-place rewriting of `binding` rules breaks the contract with existing code.

### Check

A check mechanically verifies one rule. Scoped to a single rule, not a feature.

**Frontmatter:**

```yaml
id: check-task-reschedule-atomic-edits
type: check
status: enforced
method: automated
verifies: [spec-task-reschedule]
runner: tests/list/reschedule.test.ts
```

**The regression substrate.** The set of enforced checks across all binding rules constitutes the regression suite by construction — not a per-feature test plan. Retiring a rule retires its check; adding a rule adds a check. A check must actually exercise the rule it verifies; a check that passes without exercising the rule is worse than no check, because it gives false confidence.

**Lifecycle:** `draft → enforced → retired`.

### Task

The execution primitive. One markdown file per task under `planning/tasks/<state>/<id>.md` on the `plans` branch. The folder name *is* the state.

**Task states:**

| State | Meaning | Satisfies `blocked_by`? |
|---|---|---|
| `drafted/` | sketched but not yet scheduled | no |
| `ready/` | claimable; agents looking for work list this folder | no |
| `blocked/` | has `blocked_by` entries not yet `done`/`verified`; `/reconcile` promotes when deps resolve | no |
| `claimed/` | an agent is working on it; frontmatter records `agent` and `claimed_at` | no |
| `done/` | completed, awaiting verification by a different agent | **yes** |
| `verified/` | completion confirmed by a different identity | **yes** |
| `abandoned/` | a claim was dropped; `/reconcile` returns to `ready/` or `blocked/` | no |
| `retired/` | withdrawn without completion; history only | no |

**Frontmatter and body:**

```markdown
---
id: t-implement-task-reschedule
type: task
task_type: implement
initiative: init-daily-review-pilot
updates_spec: spec-task-reschedule
status: claimed
agent: claude-a1
claimed_at: 2026-05-12T14:30:00Z
blocked_by: []
---

# Implement task reschedule

Initial brief.

## Claim notes (claude-a1, 2026-05-12)

Agent working notes…
```

**Required frontmatter:** `id`, `type: task`, `task_type`, `initiative` (parent), `status`. `updates_spec` is set only when the task creates or edits a spec. `blocked_by` references task ids across any folder.

**`task_type` → skill mapping:** `draft-bet`, `draft-initiative`, `draft-spec`, `update-spec`, `implement`, `write-bet-insight`, `write-initiative-insight`, `retro`, etc. If a task doesn't map to an existing skill, it belongs in an issue tracker, not the task queue.

**State transitions are file moves** (`git mv`) performed by the task-state scripts (`/claim`, `/complete`, `/verify`, `/abandon`, `/reconcile`). See [Scripts inventory](#scripts-inventory).

**Verifier identity.** `/verify` enforces that the author email of the `[task-verify]` commit differs from the `[task-complete]` commit — a different agent (or a human running `/verify --as <their-name>`) must sign off. The `stale_claim_threshold` in `config.yaml` controls how long a task can sit in `claimed/` before `/queue` surfaces it as stale.

### Insight

A distilled piece of evidence. Measures a bet or an initiative against its hypothesis. All three outcomes are valid — a refuted hypothesis is evidence that prevents doubling down on something that doesn't work.

**Frontmatter:**

```yaml
id: insight-daily-review-r1
type: insight
status: preliminary
title: Daily-review pilot outcome
category: analytics
measures_initiative: init-daily-review-pilot
# optionally: measures_bet: bet-daily-review
result: inconclusive                # confirmed | refuted | inconclusive
```

**Body:** summary, implication, links to raw data.

**Lifecycle:** `preliminary → supported → established → retired`. A `preliminary` insight is written when the initiative completes; humans mature it to `supported` or `established` as evidence firms up.

**Where insights live.** `direction/insights/` on `main`. Insights accumulate forever as the durable evidence ledger.

**What insights don't do.** Insights never change spec rules directly. An insight triggers a *decision* — draft a new initiative, retire an existing one, adjust a bet — made by humans.

**Specs don't carry hypotheses, so insights don't measure specs directly.** Drift is the spec's "did the rules hold?" signal; insights measure the claims attached to bets and initiatives.

Time-boxed investigations that produce an insight instead of code are drafted as [discovery-type initiatives](#initiative).

## Lifecycles (summary)

Compact reference across all artifacts. See the individual artifact sections for the meaning of each state.

| Artifact | Lifecycle |
|---|---|
| Narrative / architecture / principles | `draft` → `stable` → `retired` |
| Blueprints | generated (no lifecycle) |
| Bet | `backlog` → `planned` → `active` → `completed` |
| Initiative | `backlog` → `planned` → `active` → `completed` *(separate type from bet)* |
| Story | `draft` → `validated` → `addressed` → `retired` |
| Spec | `draft` → `accepted` → `retired` *(acceptance preconditions: see [Spec](#spec))* |
| Rule (inline / shared) | `proposed` → `binding` → `retired` |
| Check | `draft` → `enforced` → `retired` |
| Insight | `preliminary` → `supported` → `established` → `retired` |
| Task | `drafted` → `ready` \| `blocked` → `claimed` → `done` → `verified` \| `abandoned` \| `retired` |
| Skills | `draft` → `stable` → `retired` |

Only `done` and `verified` count as complete for task `blocked_by` resolution.

### Extraction pattern

Structure grows in response to real needs — don't create layers speculatively.

| When this happens | Extract this |
|---|---|
| Same story in two initiatives | Story file in `planning/stories/` |
| Same rule in two specs | Shared rule in `shared/rules/` |
| Generated blueprint insufficient for coordination | Curated blueprint in `direction/blueprint.md` |
| Three or more specs in one domain | Domain subfolder with its own checks |
| Standards repeating across specs | Principles file in `direction/principles/` |

## Operating the system

### Git coordination

Git is not just storage — it is the coordination substrate. Every task state change is a commit; every commit message carries structured metadata that scripts and humans can replay. Procedural details (worktree setup, sync flow, cross-ownership changes) live in the [`git-coordination` skill](../.claude/skills/git-coordination/SKILL.md); this section sketches the model.

#### Two branches: `main` and `plans`

- **`main`** — the product + direction branch. Owns specs, code, narrative, north-star, architecture, principles, and insights (`direction/insights/`). Work happens on feature branches off `main`; PRs merge back.
- **`plans`** — the orchestration ledger. Owns bets (`planning/bets/<state>/<id>.md`), initiatives (`planning/initiatives/<state>/<id>.md`), and tasks (`planning/tasks/<state>/<id>.md`) — folder-per-state across all three. No spec drafts, no code, no insights ever land here.

Bootstrap: `bun run setup-worktree`. Orchestration scripts refuse to run state transitions on any other branch.

#### Typical task flow

1. Agent claims: `/claim <id>` → `[task-claim]` commit on `plans`.
2. Agent switches to the right worktree and cuts a feature branch:
   - For bet/initiative drafts: off `plans` (feature branch in `<repo>-plans/`).
   - For specs/code/insights: off `main` (feature branch in `<repo>/`).
3. Agent drafts / implements on the feature branch, opens a PR into the matching target branch.
4. A different agent (or a human) reviews and merges the PR.
5. Agent switches back to the `plans` worktree: `/complete <id>` → `[task-complete]` on `plans`.
6. A different agent: `/verify <id>` → `[task-verify]` on `plans`.

`plans` is **never merged into `main`**. `main` can be merged into `plans` via `bun run sync-from-main` from the plans worktree when plans-side references to specs or insights get stale.

A pre-commit hook at `process/hooks/pre-commit` enforces the orchestration-only rule on `plans` (activated by `git config core.hooksPath process/hooks`, which `setup-worktree` runs).

#### Developer ergonomics: plans worktree

`bun run setup-worktree` creates the `plans` branch (if missing), pushes it with `-u`, adds a worktree at `../<repo-name>-plans/`, configures cone-mode sparse-checkout (materializing `planning/`, `specs/` read-only, `direction/insights/` read-only, `process/scripts/`, `process/hooks/`, root files), and registers a `merge.ours.driver` in local git config.

Day-to-day:

- `<repo-name>/` — on `main`, for specs, code, direction, and insights.
- `<repo-name>-plans/` — on `plans`, for drafting bets/initiatives and running `/claim`, `/complete`, `/verify`, `/abandon`, `/reconcile`.
- Swap by `cd`, not by `git checkout`.
- `git fetch` in either updates both.

**Branch divergence is expected.** Bets, initiatives, and task state files only exist on `plans`; specs, code, direction, and insights only exist on `main`. `sync-from-main` snapshots orchestration paths before merging and restores any bet/initiative/task file that a main-side delete would drop.

#### Commit message conventions

- `[task-claim] <task-id> by <agent>`
- `[task-complete] <task-id> by <agent>`
- `[task-verify] <task-id> by <verifier> (completed by <agent>)`
- `[task-verify-fail] <task-id> by <verifier> (completed by <agent>): <note>`
- `[task-abandon] <task-id> by <agent> (claim-from <original-agent>)[: <reason>]`
- `[task-reconcile] <n> transitions (<summary>)`
- `[task-draft] <task-id>` (manual)
- `[task-retire] <task-id>` (manual)

#### Verifier identity

Verification requires a different **git identity** — the author email of the `[task-verify]` commit must differ from the author email of the corresponding `[task-complete]` commit. `/verify` enforces this via `git config user.email` and `git log --grep` lookup.

Humans can verify: a human reviewing an agent's completion runs `/verify <id> --as <their-name>`; their git email differs from the agent's, so the check passes.

### Scripts inventory

Every script is a slash command wrapping a file under `process/scripts/`. Grouped by purpose.

#### Validation and reporting

- **`/validate`** — Frontmatter, link resolution, lifecycle states. Bet: hypothesis + kill criteria required for `active`. Initiative: `bets: []` non-empty, `initiative_type` required, type-specific fields enforced (product stories post-traction, architecture refs, fix `reveals_gap_in`). Spec: no hypothesis, no `spec_type`, no back-reference to initiative. Task: `initiative` required; `task_type` present. Runs from either worktree.
- **`/drift`** — Compares generated code against accepted specs. Flags unrecorded drift; recorded deviations report separately as debt.
- **`/coverage`** — Active bets with no measuring insight. Completed initiatives with no measuring insight. Validated stories not addressed by any initiative. Accepted specs with no checks. Fix initiatives clustering on the same target spec. Unresolved retro entries older than 30 days.
- **`/queue`** — Validates and reports on task queues. Flags unknown initiative references, unknown spec references in `updates_spec`, stale claims, cycles, dep-state mismatches.
- **`/graph`** — Renders the dependency graph (bet → initiative → spec) and the blueprint view (from accepted specs' journey tags).
- **`/indexes`** — Regenerates `INDEX.md` from frontmatter. Run after any status change or add/retire.

#### Task state transitions

All run on the `plans` branch and fetch, commit with structured metadata, and push. Races surface as push rejections.

- **`/work`** — Lists the ready queue across all initiatives. Starting point when looking for work.
- **`/claim <id>`** — Move a `ready/` task into `claimed/` for the current agent.
- **`/complete <id>`** — Move a `claimed/` task into `done/`; awaits verification.
- **`/verify <id>`** — Move a `done/` task into `verified/`. Enforces different-identity rule.
- **`/abandon <id>`** — Drop a claim; task returns to `ready/` or `blocked/`.
- **`/reconcile`** — Flip `blocked/` ↔ `ready/` based on dependency resolution; reactivate abandoned tasks.

#### Bootstrap and sync

- **`bun run setup-worktree`** — First-time setup: creates `plans` branch, adds sibling worktree, configures sparse checkout, registers merge driver, installs hooks.
- **`bun run sync-from-main`** — From the plans worktree: snapshots orchestration paths, merges `main` into `plans`, restores any bet/initiative/task file that a main-side delete would have dropped.

#### Why scripts and not agents

Every script above is deterministic. Running a state transition as a script costs nothing and cannot hallucinate a claim commit, mislink a spec reference, or mis-order a verify. Running the same logic as agent prose costs tokens, time, and introduces nondeterminism for no payoff. When a new capability feels repetitive or purely mechanical, it belongs here, not in a skill.

### Agent entry protocols

#### When given a specific task

1. Load always-read context (narrative, north-star, config).
2. Read the task; check `task_type`, `initiative`, optional `updates_spec`.
3. Load context triggered by the initiative (its bets, its `specs_addressed`).
4. Load the skill triggered by `task_type`.
5. Do the work; parallelize where independent.
6. Run the relevant script before committing (`/validate` for bet/initiative/spec changes; `/drift` for code changes).
7. Run `/complete <task-id>` when acceptance criteria are met.
8. Surface the `done` task for a different agent to verify.
9. If friction or insight occurred, load the [`retro`](../.claude/skills/retro/SKILL.md) skill.

#### When looking for work

1. Load always-read context.
2. Run `git fetch` so you see other agents' in-flight claims.
3. Run `/work` to see the ready queue across all initiative task queues.
4. Claim with `/claim <task-id>` — the script commits and pushes.
5. If the push fails (race), pull with rebase, re-check the task.
6. Follow the "given a specific task" protocol for the claimed task.

#### When asked to verify

1. Load always-read context.
2. Read the initiative's hypothesis and the task's acceptance criteria.
3. Inspect the actual work — code, commits, generated checks, spec changes.
4. Run `/validate` and any relevant checks.
5. If acceptance criteria are genuinely met: `/verify <task-id> --as <your-agent-id>`.
6. If not: add a retro entry describing the gap, and either leave `done` for the completing agent to address or `abandon` it.

Agents never browse `INDEX.md` looking for work — they check task queues via `/work`.

### Review cadence

When an initiative's pilot completes, the insight loop closes:

1. Status flips `active → completed`. A `preliminary` insight is written against the initiative's hypothesis.
2. Evidence is gathered; insight matures to `supported` or `established` (human flip).
3. A bet-level insight is written if the initiative's evidence materially moves the bet's hypothesis one way or the other.
4. A retro log entry captures what the operating model did or didn't serve during the pilot.
5. The next initiative is drafted against the (possibly-updated) bet.

Initiative insights answer "did this pilot pay off?" Bet insights answer "is the strategic claim holding up?" Both inform what we do next.

## Derivation and accountability

### Code derives from specs

"Code is derived from specs" admits several implementations:

- **Full code generation** — `drift` enforces match between generated code and specs.
- **Partial scaffolding** — specs generate module skeletons, test fixtures, or API contracts; the rest is human-written.
- **Tests and checks generated from specs** — production code written by humans, verified against generated checks.
- **Loose derivation** — specs are authoritative reference, humans write code that matches, and `drift` compares asserted behavior.

All are valid. The common requirements, regardless of derivation strategy:

1. When code and spec disagree, the spec is updated first or the deviation is recorded — never the code alone, silently.
2. Every binding rule has at least one enforced check, so the regression suite is persistent and complete by construction.

Checks carry the regression contract; the derivation strategy only determines who writes the code.

### Hypothesis accountability

Active bets and initiatives have hypotheses. An insight measures the outcome as `confirmed`, `refuted`, or `inconclusive`. `/coverage` reports bets and initiatives that have been `completed` without a measuring insight. All three outcomes are valid. A refuted hypothesis prevents doubling down on something that does not work.

**Insights never change spec rules directly.** An insight triggers a decision — draft a new initiative, retire an existing one, adjust a bet — made by humans.

## Evolution

### Legacy and evolution

| Situation | What to do |
|---|---|
| Retiring a page | Set `status: retired`, add `replaced_by`. `/validate` warns if non-retired pages reference it. |
| Renaming | Change filename and `id`, add `supersedes: old-id`. |
| Curating a blueprint | When the generated view isn't rich enough, create a curated `blueprint.md`. Keep specs' `journey` phase tags accurate. |
| Adding a domain | Create the folder. No ceremony. |
| Flipping modes | Change `mode:` in `config.yaml` once you have real users and baseline data. |

### Success criteria

The operating model is working when:

- Every active bet has a hypothesis with direction, kill criteria, and at least one metric.
- Every active initiative has a falsifiable hypothesis, success criteria, and kill criteria.
- Every accepted spec has narrative + binding rules + enforced checks and a clean `/drift`.
- Every completed initiative has an insight recording confirmed, refuted, or inconclusive.
- Every validated story is addressed by at least one initiative.
- `/validate` passes on every merged PR.
- `/drift` is clean, or drift is recorded as tracked debt.
- The retro log is reviewed when friction clusters.
- A new team member reads narrative, north-star, one bet, and one initiative, and can review a PR.
- Multiple agents work on independent initiatives concurrently without coordination failures.
- Agents can self-serve from the task queues without human intervention per task.
- Humans and agents read the same artifacts without retranslation.
- The operating model tightens over time through retro-driven iteration.
